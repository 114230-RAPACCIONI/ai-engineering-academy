# User Stories

> "A good product listens before it builds."

---

# Introduction

Este documento define las historias principales de usuario que representan las necesidades y expectativas dentro de Project ZUZU.

Las historias están basadas en las personas definidas en ART-003 y el recorrido definido en ART-004.

El objetivo no es describir funcionalidades.

El objetivo es comprender qué valor necesita recibir cada usuario.

---

# User Story Format

Todas las historias siguen el formato:
Como [usuario]

quiero [acción]

para [beneficio]


---

# Epic 1 — User Identity

## US-001 — Crear perfil personal

### User Story

Como usuario nuevo,

quiero crear mi perfil dentro de Project ZUZU,

para que la plataforma pueda comprender mi contexto y objetivos.

---

### Acceptance Criteria

- El usuario puede registrarse.
- Puede indicar experiencia previa.
- Puede definir objetivos.
- Puede actualizar información.

---

# US-002 — Definir objetivo de aprendizaje

### User Story

Como usuario,

quiero definir qué quiero lograr,

para recibir una experiencia alineada con mis necesidades.

---

### Acceptance Criteria

El usuario puede seleccionar:

- aprender programación;
- mejorar como desarrollador;
- incorporar IA;
- construir productos.

---

# Epic 2 — Learning Journey

## US-003 — Visualizar mi camino

### User Story

Como usuario,

quiero visualizar mi recorrido de aprendizaje,

para saber dónde estoy y cuál es mi próximo paso.

---

### Acceptance Criteria

El sistema muestra:

- etapa actual;
- próximos objetivos;
- progreso.

---

# US-004 — Completar desafíos

### User Story

Como usuario,

quiero realizar desafíos prácticos,

para transformar conocimiento en capacidad.

---

### Acceptance Criteria

Un desafío debe incluir:

- objetivo;
- contexto;
- resultado esperado;
- feedback.

---

# Epic 3 — Knowledge System

## US-005 — Consultar conocimiento

### User Story

Como usuario,

quiero acceder a conceptos organizados,

para comprender fundamentos tecnológicos.

---

### Acceptance Criteria

El conocimiento debe:

- estar organizado;
- tener relaciones;
- permitir búsqueda.

---

# US-006 — Entender antes de aplicar

### User Story

Como usuario,

quiero comprender por qué una solución funciona,

para poder tomar mejores decisiones.

---

### Acceptance Criteria

La plataforma debe priorizar:

- explicación;
- contexto;
- ejemplos;
- razonamiento.

---

# Epic 4 — AI Collaboration

## US-007 — Conversar con un mentor IA

### User Story

Como usuario,

quiero interactuar con un asistente inteligente,

para recibir acompañamiento durante mi aprendizaje.

---

### Acceptance Criteria

El mentor debe:

- entender contexto;
- explicar conceptos;
- hacer preguntas;
- guiar decisiones.

---

# US-008 — Recibir feedback sobre soluciones

### User Story

Como usuario,

quiero recibir revisión de mis ideas y proyectos,

para mejorar mi capacidad técnica.

---

### Acceptance Criteria

La IA debe analizar:

- decisiones;
- arquitectura;
- calidad;
- posibles mejoras.

---

# Epic 5 — Project Workspace

## US-009 — Crear un proyecto

### User Story

Como usuario,

quiero crear proyectos reales,

para aplicar lo aprendido.

---

### Acceptance Criteria

Un proyecto debe permitir:

- descripción;
- objetivos;
- documentación;
- progreso.

---

# US-010 — Documentar decisiones

### User Story

Como usuario,

quiero registrar decisiones técnicas,

para desarrollar hábitos profesionales.

---

### Acceptance Criteria

Debe permitir documentar:

- problema;
- alternativas;
- decisión;
- consecuencia.

---

# Epic 6 — Progress

## US-011 — Visualizar evolución

### User Story

Como usuario,

quiero ver mi progreso,

para entender mi crecimiento.

---

### Acceptance Criteria

Debe mostrar:

- habilidades;
- proyectos;
- avances.

---

# US-012 — Recibir recomendaciones

### User Story

Como usuario,

quiero recibir sugerencias personalizadas,

para saber qué mejorar después.

---

### Acceptance Criteria

Las recomendaciones deben considerar:

- progreso;
- objetivos;
- proyectos.

# Non Goals

Inicialmente no buscamos:

- crear una red social;
- competir con plataformas educativas masivas;
- crear un marketplace;
- gamificar excesivamente.

---

# Final Statement

Las historias de usuario de Project ZUZU representan una idea central:

El producto no existe para enseñar contenido.

Existe para acompañar una transformación.
